## Remaining Risks

- `bot.py` still contains duplicated legacy sections. The app is working, but future refactors should continue moving logic into `services/` in smaller slices.
- Stripe webhook idempotency is now present, but production should verify Railway logs with a real Stripe test event.
- Admin pages are functional and private, but deeper RBAC edit screens should be expanded after the core billing/account flows are stable.
- Local SQLite works for early launch, but production SaaS growth should move to PostgreSQL.

## Recommended Next Infrastructure Upgrades

- Move from SQLite to PostgreSQL using `DATABASE_URL` for multi-worker safety and backup durability.
- Split the Telegram bot, web app, and background jobs into separate Railway services once traffic grows.
- Add Redis or another shared cache for market data, rate limits, and background jobs.
- Add external error monitoring and uptime checks.
- Add scheduled backups and a tested restore process.
